# 软件移植提示词模板

这个文件是一个可复用的提示词模板，用来让 AI 帮你把软件移植到 Seele OS。

你可以复制下面的模板，替换其中的占位符后直接发给 AI。

```md
我现在要把一个软件移植到我的操作系统 Seele OS。

我的工作流是这样的：
1. 先把上游源码 clone 到 `porting_dev/<pkgname>/`。
2. 在源码目录里单独写一个 `Makefile.seele`，用于移植和测试构建。
3. 使用 `Makefile.seele` 反复修改和调试，直到这个软件可以在 Seele OS 上成功构建，或者至少推进到一个清晰的阶段。
4. 移植完成后，生成需要放进 packages 系统里的 patch。
5. 最后在我的 packages 系统中补上正式的 package makefile 和 patches。

重要约束：
- `Makefile.seele` 只是移植阶段和测试阶段使用的 makefile，不是最终放进 packages 系统的 makefile。
- 最终结果必须能融入我现有的 packages 系统目录结构和风格。
- 优先做小而清晰、便于审查的修改，不要做大而失控的改动。
- 不要覆盖与当前任务无关的本地修改。
- 在搜索源码树时，优先使用 `rg` 之类的快速工具。

请帮我移植这个软件：`<software-name>`

源码位置：
- 上游源码目录：`porting_dev/<pkgname>/`

参考：
- 你应该先查看 `porting_dev/bash/Makefile.seele`，并优先参考它的整体风格；除非当前目标软件明显需要不同的构建方式。

我希望你做的事情：
1. 先检查源码树，判断这个软件使用的构建系统是什么。
2. 创建或修改 `porting_dev/<pkgname>/Makefile.seele`，让它能够用于在 Seele OS 上构建这个软件。
3. 在合适的情况下，使用 Seele 的交叉工具链和我现有的 OS 构建约定。
4. 如果软件源码本身需要修改，请只做最小必要修改，并让这些修改适合后续生成 patch。
5. 根据构建报错持续迭代，尽可能把问题修掉，直到构建成功，或者至少推进到一个合理的停止点。
6. 当移植稳定后，整理出需要放进 packages 系统的 patch 文件。
7. 为我的 packages 系统补上最终的 package makefile 和 patch 布局。
8. 最后总结：
   - `Makefile.seele` 做了哪些改动
   - 需要哪些源码 patch
   - 新增了哪些 package 文件
   - 还剩下哪些阻塞问题，或者哪些运行时验证还没做

构建前提：
- 目标系统：Seele OS
- 常见目标三元组：`x86_64-unknown-none`
- 移植时通常会使用仓库中的 relibc
- 整体构建风格应尽量和我现有的 ports 保持一致

工作规则：
- 编辑之前先检查相关已有文件，不要先入为主。
- 在修改文件之前，先说明你准备改什么。
- 不要只给建议；请直接在工作区里做实际修改。
- 最终回答保持简洁，并带上关键文件路径。

如果有信息缺失，请基于仓库现有结构和已有示例做保守推断，不要过早停下来等待。
```

建议用法：
- 发送给 AI 之前，先把 `<software-name>` 和 `<pkgname>` 替换掉。
- 如果目标软件不是 Autotools 项目，可以在提示词里明确要求 AI 按实际构建系统调整方案。
- 如果你已经知道这个软件有特殊需求，可以在提示词末尾补一个“项目特殊说明”小节。
